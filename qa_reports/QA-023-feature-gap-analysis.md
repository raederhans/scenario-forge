## 三、功能方向一：战略标注工具

### 3.1 需求分析

大战略游戏玩家最核心的创作需求之一是在地图上"讲故事"——不只是展示"谁拥有哪里"，而是展示"发生了什么"。

典型使用场景：

- 巴巴罗萨行动进攻路线图（多条弯曲箭头 + 包围圈标注）
- 一战前线演变图（锯齿状前线 + 战役位置标记）
- 架空历史剧本说明图（区域高亮 + 文本说明 + 箭头指向）
- 贴吧/Reddit AAR（After Action Report）配图

目前的创作流程断裂：用户在 Map Creator 中完成底图 → 导出 PNG → 在 Photoshop/Figma/Paint 中添加箭头和文字 → 发布。标注工具可以消除这个流程断裂。

### 3.2 功能子模块设计

#### A. 箭头工具

| 属性 | 说明 |
| --- | --- |
| 绘制方式 | 点击放置控制点，自动生成贝塞尔曲线 |
| 箭头样式 | 实心三角、空心三角、双箭头、无箭头 |
| 线条样式 | 实线、虚线、点线 |
| 颜色 | 自定义颜色 + 从当前调色板选择 |
| 宽度 | 可调节（1-10px） |
| 预设类型 | 进攻箭头（红）、撤退箭头（蓝）、补给线（绿虚线）、海上运输（蓝虚线） |

#### B. 前线线条

| 属性 | 说明 |
| --- | --- |
| 绘制方式 | 点击放置节点，生成锯齿/波浪状线条 |
| 线条风格 | HOI4 风格锯齿、平滑曲线、战壕线 |
| 颜色 | 双色（攻方/守方） |
| 填充 | 可选半透明区域填充（表示控制区） |

#### C. 标记/图标

| 属性 | 说明 |
| --- | --- |
| 预设图标 | 战役（爆炸）、围城、海战、空袭、首都星号、据点 |
| 放置方式 | 点击地图放置 |
| 大小/旋转 | 可调节 |
| 自定义图标 | 允许上传 SVG/PNG 作为自定义标记 |

#### D. 文本标注

| 属性 | 说明 |
| --- | --- |
| 放置方式 | 点击地图位置，输入文本 |
| 字体 | 预设字体系列（衬线/无衬线/手写体） |
| 大小/颜色 | 可调节 |
| 背景 | 可选半透明背景框 |
| 旋转 | 可沿路径旋转（用于标注河流/山脉名称） |

#### E. 区域高亮

| 属性 | 说明 |
| --- | --- |
| 绘制方式 | 矩形选框 / 椭圆 / 自由多边形 |
| 样式 | 虚线边框 + 半透明填充 |
| 标签 | 可附加说明文字 |

### 3.3 技术可行性

**现有基础设施：**

- SVG 叠加层（`#map-svg`）已用于特殊区域绘制和 hover 高亮
- `special_zones` 绘制已实现 vertex-by-vertex 多边形编辑（Start/Cancel/Finish/Undo vertex）
- D3.js v7 原生支持 SVG path、marker、text 渲染
- 投影变换已就位（地理坐标 ↔ 屏幕坐标）

**实现路径：**

```
新增模块: js/ui/annotation_toolbar.js
  ├── ArrowTool: 贝塞尔曲线 + SVG marker-end
  ├── FrontlineTool: 扩展现有 special zone 绘制逻辑
  ├── MarkerTool: SVG <use> + 预设 symbol 库
  ├── TextTool: SVG <text> + foreignObject (rich text)
  └── HighlightTool: SVG <rect>/<ellipse>/<polygon>

数据模型: 扩展 Project JSON
  annotations: [
    { type: "arrow", points: [[lon,lat],...], style: {...} },
    { type: "frontline", points: [...], style: {...} },
    { type: "marker", position: [lon,lat], icon: "battle", ... },
    { type: "text", position: [lon,lat], content: "...", ... },
    { type: "highlight", geometry: {...}, label: "..." }
  ]

渲染层: 在 SVG 层中添加 annotation group
  → z-order: 在政治边界之上、UI 控件之下
  → 随 zoom/pan 变换
  → 导出时合成到 Canvas 输出
```

**与现有架构的兼容性：高**

- SVG 叠加层已有完整的 zoom/pan 同步逻辑
- 项目保存/加载已支持 JSON 序列化
- 撤销/重做系统可扩展到标注操作
- 导出合成需要在 Canvas 渲染时额外绘制 SVG 内容（`canvg` 或手动重绘）

**估算工作量：** 中等（~2-3 周核心功能，后续迭代打磨）

## 四、功能方向二：时间轴与动画系统

### 4.1 需求分析

当前 5 个剧本（未来可能 10+）是孤立的时间快照。用户无法感受到"版图在时间中流动"。

在 Reddit、B站、YouTube 上，最受欢迎的大战略/历史内容形式之一是**版图变化 timelapse**。目前这些内容的制作流程极其繁琐：逐帧截图 → 视频编辑软件 → 添加时间标签 → 手动对齐。

如果 Map Creator 能内置时间轴和动画导出，这将成为**杀手级功能**——目前市面上没有任何工具能做到这一点。

### 4.2 功能子模块设计

#### A. Timeline Slider（核心）

```
┌────────────────────────────────────────────────────┐
│  ◄  1914  ·  1918  ·  1936  ·  1939  ·  1945  ►  │
│       │                  ▲                          │
│       │              [当前位置]                      │
│  ════════════════════█████════════════════════════  │
│                      [拖动]                         │
│                                          [▶ 播放]  │
└────────────────────────────────────────────────────┘
```

| 属性 | 说明 |
| --- | --- |
| 关键帧 | 每个剧本是一个关键帧节点 |
| 拖动 | 拖动滑块，在两个关键帧之间插值 |
| 播放 | 自动播放动画（可调速度） |
| 关键帧标签 | 显示年份和剧本名称 |
| 当前日期显示 | 地图上叠加当前年份/日期 |

#### B. 跨剧本过渡动画

| 过渡类型 | 说明 | 技术方案 |
| --- | --- | --- |
| **硬切** | 直接切换（现有行为） | 无需额外工作 |
| **淡入淡出** | 旧颜色淡出，新颜色淡入 | Canvas alpha 插值 |
| **边界渐变** | 边界线逐渐移动到新位置 | Feature 归属插值 + 重新着色 |
| **涟漪扩散** | 从变化区域中心向外扩散 | 按距离延迟的 feature 切换 |

**核心技术挑战——跨剧本插值：**

```
场景 A (1936): feature AFG-1741 → owner GER
场景 B (1939): feature AFG-1741 → owner GER (不变)
场景 A (1936): feature CZ_ADM2_xxx → owner CZE
场景 B (1939): feature CZ_ADM2_xxx → owner BOM (变了)

插值策略:
  t=0.0: 显示场景 A 的完整归属
  t=0.0~1.0: 仅"变化"的 feature 执行颜色过渡
  t=1.0: 显示场景 B 的完整归属
```

**不需要几何插值**——边界不需要平滑移动，只需要 feature 归属在两个状态之间过渡（颜色渐变即可）。这大大降低了实现难度。

#### C. Before/After 对比

| 模式 | 说明 |
| --- | --- |
| **分割线** | 垂直/水平分割线，左边显示剧本 A，右边显示剧本 B |
| **叠加闪烁** | 交替显示两个剧本，高亮变化区域 |
| **差异高亮** | 显示一个剧本，但用不同颜色标记"变化了的 feature" |

#### D. 动画导出

| 格式 | 说明 | 技术方案 |
| --- | --- | --- |
| **GIF** | 适合 Reddit/Discord 分享 | Canvas 逐帧捕获 → gif.js 编码 |
| **WebM/MP4** | 高质量视频 | MediaRecorder API 或 ffmpeg.wasm |
| **帧序列** | PNG 序列，用户自行合成 | Canvas.toDataURL 逐帧导出 |

### 4.3 技术可行性

**现有基础设施：**

- `scenario_manager.js` 已实现完整的剧本加载/切换（`loadScenarioBundle()`, `applyScenarioById()`）
- 每个剧本的 `owners.by_feature.json` 提供 feature→owner 映射
- 颜色系统（`color_manager.js`）支持任意颜色赋值
- Canvas 渲染管线支持 `requestAnimationFrame` 调度

**关键实现挑战：**

| 挑战 | 难度 | 解决方案 |
| --- | --- | --- |
| 跨剧本 feature ID 对齐 | 中 | 使用共享的 `properties.id`（所有剧本都基于同一拓扑） |
| 同时加载多个剧本数据 | 中 | 预加载相邻关键帧的 owners 数据到内存 |
| 不同剧本拓扑不同 | 高 | TNO 有自定义拓扑（12,829 vs 11,326 features）——需要降级处理或排除 |
| 动画帧率 | 中 | Canvas 渲染足够快（当前已有 rAF 调度） |
| GIF 编码性能 | 中 | 使用 Web Worker + gif.js 异步编码 |

**核心限制：** TNO 1962 使用独立拓扑，无法与其他剧本平滑插值。可以支持的无缝时间线为 `1914 → 1918 → 1936 → 1939 → 1945 → 1991 → modern`（这些都共享同一基础拓扑）。

**实现路径：**

```
Phase 1: Timeline UI + 硬切
  → 底部 timeline 控件
  → 点击关键帧切换剧本
  → 播放按钮自动切换

Phase 2: 颜色过渡动画
  → 预加载相邻剧本 owners 数据
  → 计算 diff（哪些 feature 归属变化了）
  → 对变化 feature 执行颜色插值动画

Phase 3: Before/After + 差异高亮
  → 分割线对比视图
  → 变化 feature 高亮标记

Phase 4: 动画导出
  → GIF / WebM / 帧序列导出
```

**估算工作量：** Phase 1 约 1 周，Phase 2 约 2 周，Phase 3-4 各约 1 周。

### 4.4 竞品分析

目前市面上**没有**现成工具能做到跨历史时期的版图动画过渡：

- **HistoricalMapChart.com**: 仅静态地图，无动画
- **MapChart.net**: 手动涂色，无场景/时间线概念
- **GeaCron**: 有时间轴但 UI 过时且不支持自定义
- **HOI4 Timelapse 视频**: 都是手工用视频编辑软件逐帧拼接

这是一个明确的**蓝海功能**。

## 五、功能方向三：视觉增强系统

### 5.1 国旗/徽章系统

#### 需求

国家在地图上仅以颜色区分。对于有大量小国的场景（如 TNO 1962 的俄罗斯碎片化），颜色难以区分。国旗是天然的视觉身份标识。

#### 功能设计

| 子功能 | 说明 |
| --- | --- |
| **国旗图标** | 在国家中心（或首都位置）显示缩略国旗 |
| **图例国旗** | Legend 中每个国家名称旁显示国旗缩略图 |
| **国旗数据源** | HOI4 mod 自带 `gfx/flags/` TGA 文件；可预转换为 PNG/SVG |
| **自定义旗帜** | 架空历史创作者可上传自定义国旗 |
| **显示模式** | 可选关闭（纯色地图）或按 zoom 级别自动显隐 |

#### 技术方案

```
数据层:
  data/flags/{palette_id}/{TAG}.png  (预转换的 64x42 PNG)
  → HOI4 Vanilla: gfx/flags/*.tga → ImageMagick 批量转换
  → Kaiserreich: 同上（mod 目录）
  → 自定义: 用户上传，存入 Project JSON (base64)

渲染层:
  Canvas:
    → 在每个国家质心位置绘制国旗图标
    → 按 zoom 级别控制显示/隐藏（zoom > 3 时显示）
    → 与城市点图层类似的 LOD 策略

  Legend:
    → 在国家名称前添加 <img> 元素
```

**与现有架构兼容性：高** — 城市点图层 (`city_points`) 已建立了在质心位置渲染图标 + zoom 自适应的完整模式，国旗图标可复用同一渲染路径。

**估算工作量：** 低（~1 周，主要是数据准备和渲染集成）

### 5.2 投影切换

#### 需求

Equal Earth 是优秀的等面积投影，但不同场景和用途需要不同投影：

| 投影 | 适用场景 |
| --- | --- |
| Equal Earth（当前） | 通用政治地图 |
| Mercator | 游戏风格（HOI4/EU4 使用类 Mercator） |
| Orthographic | 地球仪视图、区域聚焦 |
| Azimuthal Equal Area | 极地视角、冷战对峙视图 |
| Conic Conformal | 区域地图（欧洲/北美聚焦） |

#### 功能设计

| 子功能 | 说明 |
| --- | --- |
| **投影选择器** | 下拉菜单选择投影类型 |
| **旋转控制** | Orthographic/Azimuthal 投影支持拖动旋转中心 |
| **区域预设** | "聚焦欧洲"、"聚焦太平洋"等快捷视图 |
| **投影动画** | 切换投影时平滑过渡（D3 投影插值） |

#### 技术方案

D3.js 原生支持投影切换和插值：

```javascript
// 已有
const projection = d3.geoEqualEarth();

// 扩展
const projections = {
  equalEarth: d3.geoEqualEarth(),
  mercator: d3.geoMercator(),
  orthographic: d3.geoOrthographic().rotate([-10, -50]),
  azimuthal: d3.geoAzimuthalEqualArea().rotate([-10, -50]),
  conicConformal: d3.geoConicConformal().parallels([43, 62])
};

// 投影切换动画
d3.transition()
  .duration(1000)
  .tween("projection", () => {
    const interp = d3.geoProjection(
      d3.geoInterpolate(oldProjection, newProjection)
    );
    return t => { projection = interp(t); render(); };
  });
```

**关键影响点：**

- `map_renderer.js` 中的 `projection` 引用需要从常量变为可切换
- 空间索引（hit testing grid）需要在投影切换后重建
- 海岸线 LOD 缓存需要按投影维度扩展
- Orthographic 投影需要处理背面裁剪（globe back-face culling）

**估算工作量：** 中等（~2 周，投影本身简单，但空间索引/LOD 缓存的适配需要仔细处理）

### 5.3 数据可视化层（Choropleth）

#### 需求

超越分类着色（红=德国，蓝=法国），支持按数值着色（深红=人口1亿，浅红=人口1000万）。

这不是做通用 GIS，而是面向目标用户的**叙事增强**：

- "1936年各国工厂数量分布"
- "一战后领土变化百分比"
- "冷战两大阵营军事力量对比"

#### 功能设计

| 子功能 | 说明 |
| --- | --- |
| **数据导入** | 上传 CSV/JSON（列：country_tag, value） |
| **预设数据集** | 内置历史数据（人口、面积、HDI 等） |
| **色阶选择** | Sequential（单色渐变）、Diverging（双极）、Categorical（分类） |
| **图例** | 自动生成数值色阶图例 |
| **交互** | hover 显示具体数值 |
| **比较模式** | 两个数据集的差值可视化 |

#### 技术方案

```
数据模型:
  choroplethData: {
    name: "1936 Population (millions)",
    source: "user_upload" | "builtin",
    values: { "GER": 69.3, "FRA": 41.5, "SOV": 168.5, ... },
    colorScale: "sequential_reds",
    domain: [0, 200]
  }

渲染:
  → 替换 ColorManager 的 featureColors 输出
  → 使用 d3.scaleSequential(d3.interpolateReds).domain([0, 200])
  → Feature → owner_tag → value → color
  → 生成数值 Legend（连续色阶条）

预设数据集:
  data/datasets/population_1936.json
  data/datasets/population_1914.json
  data/datasets/gdp_modern.json
  → 按 country_tag 索引的简单 key-value JSON
```

**与现有架构兼容性：高** — 颜色系统 (`color_manager.js`) 已经是 feature→color 的映射管线，Choropleth 只是换一种映射策略。

**估算工作量：** 中等（~2 周，核心逻辑简单，但数据导入 UI 和图例需要打磨）

## 七、与现有产品路线图的关系

### 7.1 对 P0 的影响

### 7.2 对 P1 的影响

投影切换、数据可视化、存档导入属于 P2 "平台扩展"范畴，不应分散 P0/P1 的注意力。

## 八、关键技术风险

| 风险 | 影响功能 | 缓解方案 |
| --- | --- | --- |
| SVG 标注在高 zoom 时性能下降 | 战略标注 | 超过阈值时将 SVG 标注栅格化到 Canvas |
| 跨剧本 feature ID 不一致 | 时间轴动画 | 使用 `properties.id` 作为稳定键；TNO 拓扑不参与插值 |
| 国旗数据版权 | 国旗系统 | 使用开源国旗素材（如 flagpedia.net SVG）或从游戏 mod 提取 |
| 投影切换导致空间索引失效 | 投影切换 | 投影切换后延迟重建空间索引（用 `requestIdleCallback`） |
| 动画导出内存溢出 | 时间轴导出 | 使用流式 GIF 编码器；限制最大帧数 |

## 九、结论

Map Creator 作为"场景化地图工作台"的核心能力已经很强。**战略标注工具**和**时间轴系统**是两个能将产品从"静态地图编辑器"提升为"动态地缘叙事工具"的关键功能，且在目标用户群体中具有明确的蓝海地位。

建议将这两个方向纳入 P1 的第一波实施，与 Scenario Pack 系统化和新剧本扩展并行推进。
