## 已达成决议

### 一、产品定位与总体气质
- 主界面第一原则是 `地图优先`，地图必须始终是第一视觉主角。
- 主要用户按“模组作者 / 玩家”设计，不按纯专业绘图师设计。
- 产品整体气质定为 `专业克制 + 轻策略感`。
- 策略感只能作为轻气质存在，不能压过可读性、秩序感和编辑效率。

### 二、主界面结构骨架
- 右栏是唯一主工作栏，复杂编辑必须收束到这里。
- 左栏不是第二主工作栏，而是 `窄模式栏 + 可展开文字层`。
- 左栏主要承载 `模式 + 一级任务入口`，不承载深层参数编辑。
- 顶部 `scenario context bar` 只保留极轻状态表达，不再承担深编辑。
- `bottom dock` 默认收起，只在特定任务里出现。
- `bottom dock` 只承载当前模式的短动作，以及极少数通用短动作，如撤销 / 重做。
- dock 不是第二工作台，不承载需要停下来阅读、配置、总览或项目级编辑的内容。
- 浮层系统只负责短时反馈与局部补充，不再承担深编辑和长期工作区职责。
- 启动 overlay 不再占据首屏中心位置。

### 三、右栏与 Scope 规则
- 右栏不再按 `Inspector / Project` 这种平级栏目表达。
- 右栏内部改为同一工作栏中的不同 `scope`，用来表达不同编辑焦距。
- 当前认可的 `scope` 方向包括：当前对象、当前图层、当前项目。
- 空选择状态下，右栏默认落在 `项目 scope`，保证没有选中对象时仍有稳定落点。
- `scope` 切换策略定为 `自动默认 + 手动锁定`。
- 系统可以给出默认推荐，但不能持续抢夺用户控制权。
- 用户一旦手动切换到别的 `scope`，界面必须尊重这个决定。
- 手动锁定持续到“离开当前任务”为止，而不是每次点选都被重置。
- 当前讨论认可的推荐优先级，从高到低如下：
  - 用户刚刚明确表达的意图
  - 当前 `scope` 是否仍然有效
  - 当前选中的对象类型
  - 当前所处模式 / 工具
  - 系统默认落点

### 四、引导与反馈
- 引导采用双层结构：
  - 左栏负责解释当前模式是什么
  - 右栏负责解释当前任务该做什么
- 浮层只处理阻塞态、短时确认、局部提示，不再充当第二工作区。
- 新手引导和“当前该做什么”的说明，不能只堆在启动 overlay、顶部状态条或单一侧栏中。
- 反馈系统必须更统一，不能继续让 toast、hint、chip、banner、context bar 各说各话。

### 五、Typography 总方向
- 主 UI 走 `单主字族`，不走双主字族对比。
- 主字族气质走 `人文 Sans`。
- 字体来源先走 `稳定字栈`，不额外引 webfont。
- 产品按 `双语平权` 设计，英文和中文都必须像同一产品。
- 轻策略感只允许出现在 `app 名 + 主壳标题` 这种极少数壳层锚点。
- 顶层锚点的存在感主要来自 `主标题比例与字重`，不靠颜色和装饰。
- 大写 / kicker 语法只允许作为极少数壳层例外，不再作为系统习惯。
- 当前大量“小标题 / 小标签角色”要大收口。
- 表单标签 / 控件标签回到 `元信息 / 表单标签`，不再装成小标题。
- 状态文字不再单独长成一套夸张 typographic species，而是回归正文或元信息，再叠加状态语义。
- 数值 / 坐标 / 计量 / ID / 快捷键正式当成独立语言处理。
- 整体排版节奏走 `中性偏紧凑`。
- 视觉完成感走 `更平面、更结构化`，不靠更多卡片和浮起感。
- 强调色策略保留 `UI 冷色 + 地图暖色` 的双强调分工。
- 层级落差要 `清楚但克制`，不走展示页戏剧化跳跃。

### 六、5 级文字矩阵
- `顶层锚点`
  - 只留给 `app 名` 和 `主壳标题`
  - 这是唯一允许带一点轻策略感的文字层
- `区域标题`
  - 普通 section / panel / 当前任务标题
  - 负责告诉用户“这是哪一块内容”
- `正文 / 任务说明`
  - 空状态说明、操作说明、hint、真正需要读的句子
  - 负责承载解释和引导
- `元信息 / 表单标签`
  - 字段名、辅助标签、dock group label、inspector 小标签、popover label 等
  - 负责辅助理解，不再扮演标题角色
- `数值 / 代码`
  - 百分比、坐标、计量、ID、快捷键、需要对齐比较的内容
  - 这是工作站感的重要来源之一

### 七、按钮语法与控件层级总方向
- 整体按钮系统走 `文本优先`，不走全局 icon-first。
- `icon-only` 只作为明确豁免存在，不能作为默认按钮语言扩散到全产品。
- `icon-only` 当前认可的长期豁免范围包括：
  - 高频工具选择
  - 视口控制
  - 少量壳层 utility
- `语言切换` 不属于纯 icon-only 豁免项，必须保留可见文字状态。
- 每个当前工作面默认只允许 `一个真正的 primary action`。
- 当前任务里的唯一 `primary action` 默认跟随任务内容，放在 `任务段末尾`。
- `popover` 只允许承担 `局部、短停留、贴身` 的微操作职责。
- `popover` 可以在高密度视觉操作区保留，但必须严格限制数量，且不能膨胀成第二工作区。
- 全产品中表达“切换 / 当前模式 / 当前选中”的语法，默认只保留 `两种`：
  - `major tabs`
  - `compact toggle / segmented`
- 工具选中态应被视为后一类的变体，不再额外长出第三第四套切换语言。
- `Delete / Clear / Reset` 这类危险动作默认退回文字面板或明确详情区表达，不再混进顶条和主工具带。
- 对象列表、预设列表、条目列表默认不再长期暴露一整排行内按钮。
- 对象层动作默认收进单入口，列表行本身主要承担“进入当前对象任务”的职责。
- 用户点选对象 / 预设 / 条目时，默认行为是 `直接进入右栏详情`，而不是先选中再寻找单独 Edit 按钮。

### 八、壳层 utility 与任务动作总方向
- 壳层采用 `硬分家` 原则，不再把边缘一圈当成综合做事区。
- 壳层只承担 4 类职责：
  - 环境状态
  - 视口控制
  - 极少数全局 workspace 入口
  - 固定的全局帮助 / 引导入口
- 顶部 `scenario context bar` 重新定义为 `轻状态条`，不再承载任务动作。
- 顶部条允许保留 `一个固定的全局帮助 / 引导入口`，但不再保留任务工作区入口。
- `Guide` 作为壳层轻入口保留，但点击后默认切到右栏 `项目 scope` 的支持区，而不是承载完整壳层帮助系统。
- 右上 utility 区重新定义为 `环境角`，只承载环境、视口、系统状态和极少数 workspace 入口。
- 右上 utility 区的常驻可见密度按 `功能组` 计数，生产态默认不超过 `3 组`。
- 左右侧栏开合按钮继续作为贴边独立的壳层控制存在，不并入右上 utility 区。
- `Transport workbench` 保留为壳层例外入口，但从顶条移出，进入右上 utility 区。
- 除 `Transport` 外，当前讨论不认可再新增新的壳层级 workspace 入口。
- `Reference / Export / Guide` 的完整内容统一回到右栏 `项目 scope`，不再常驻壳层。
- `bottom dock` 明确定义为 `当前任务短工具带`，不再承担壳层 utility 或项目级支持工具职责。
- `bottom dock` 只保留：
  - 高频工具选择
  - 当前任务的模式切换
  - 当前任务的微动作
  - 极少数通用高频动作，如撤回 / 重做
- `bottom dock` 不再保留：
  - 项目级支持工具
  - 长停留配置
  - 危险动作
  - 导出
  - 参考入口
- 右栏继续是唯一主工作栏，并额外承担 `项目级支持工具正式落点` 的职责。
- `workspace 入口 / 支持工具 / 任务动作` 三类入口的判断标准如下：
  - `workspace 入口`：带用户进入相对独立的工作空间
  - `支持工具`：服务做任务的人，但不直接推进当前任务
  - `任务动作`：直接推进当前任务、改变当前任务状态或完成当前任务结果
- 危险动作仍属于 `任务动作` 的特殊子类，不属于壳层 utility，也不属于支持工具。

### 九、信息密度与留白节奏总方向
- 全产品的信息密度基调定为 `外松内紧`。
- `松` 不是装饰性空白，而是地图优先、壳层退后、章节清楚成立的基础条件。
- `紧` 不再作为全局默认节奏，而是受控准入，只留给真正需要高频扫读和快操作的局部专业块。
- 全产品默认只使用 `三档密度`：
  - `松`
  - `中`
  - `紧`
- 密度档位的分配原则定为 `区域身份优先`，不是状态优先。
- 地图画布与外围壳层天然偏 `松`。
- 右栏主体天然偏 `中`。
- `紧` 这一档只允许进入 `局部专业块`，不允许扩散成大章节默认态，也不允许扩散到壳层。
- 右栏章节层级默认走 `留白优先`，不走 `card 优先`。
- 大章节默认依靠更大的章节间留白、清楚的标题起伏和克制的细分隔成立，而不是依靠一层层完整 card 叠出层级。
- `card` 从默认外壳退回为少数例外，不再作为几乎所有区块的默认容器。
- 右栏默认采用 `单主章展开`，不允许多个大章节长期同时以同等声量展开。
- 右栏内部默认节奏为 `先列表，后详情`。
- 默认先给用户一个干净的扫描区，进入对象后才完整展开详情与动作，不再让列表与详情长期并列裸露。
- 帮助说明、hint、辅助文案默认 `隐一点`，只保留最必要的一句，更多解释在进入对象、触发错误、空状态或悬浮提示时再出现。
- `bottom dock` 在没有进入深入任务时默认保持 `弱存在`，不再以高声量主工具台姿态长期压在地图前方。
- 左侧 rail 默认是 `安静入口脊柱`，负责模式入口和少量状态提示，而不是另一条长期高声量的小工具带。

### 十、区域默认分档与代表性样本判断
- 区域密度落地纪律定为 `区域先有默认档，再开局部例外`。
- 地图本体默认是 `松`。
- `scenario context bar` 默认是 `松`，属于轻状态边框，不属于工具带。
- 左侧 rail 默认是 `松`，属于安静入口脊柱。
- 右上 `zoom / utility` 默认是 `中`，主身份是 `视口区`，不是全局杂项收容区。
- 右栏主体默认是 `中`。
- `project scope` 默认是 `标准中档`，不按对象细节区的高密语气处理。
- `bottom dock` 远看保持统一整体感，但默认是轻壳；真正允许进入 `紧` 的是进入任务后的局部内容。
- `Transport workbench` 是合法的 `紧` 档区域，属于局部专业工作区，不属于壳层轻区域。
- `Appearance` 在左栏中的身份只应是 `入口层`，不应继续长期维持深工作区结构。
- `紧` 这一档的默认准入范围继续收口为：
  - 对象列表
  - 局部工具组
  - 字段区 / 明细行
  - 少量专业工作区内部的局部微控件

### 十一、文字角色收口纪律
- 文字系统的收口纪律定为 `一角一义`，不再接受同名长期跨层混用。
- 强标题默认只保留给：
  - 壳层锚点
  - 当前 `scope / 当前任务`
- 左侧 rail 中，`Scenario / Appearance` 这类只属于 `入口标签`，不是完整工作区标题。
- 左侧 rail 中的 `section-header` 只作为 `入口摘要标签`，不再扮演正式章节标题。
- 右栏中的 `section-header` 只作为 `结构标签`，不再承担强标题职责。
- `section-header-block` 锁回 `元信息 / 标签层`，不再跨去做小节标题。
- `sidebar-tool-hint` 锁回 `正文式任务说明`，不再同时承担状态徽记和标题职责。
- `bottom dock` 空闲态的文字上限只到 `元信息`。
- `bottom dock` 的组结构可以常驻，但只有当前相关组的轻标签更明显，其他组不再同声量长期露出。
- `bottom dock` 中真正允许 `icon-only` 的，仍只限已确认的豁免项；其他动作不能因为放在 dock 里就顺手图标化。
- `scenario context bar` 中的状态文字统一视为 `元信息状态`。
- `scenario context bar` 只允许 `主状态突出，其余弱化`，不再所有状态项长期同声量并列。
- `scenario context bar` 中的状态项形态更接近轻文本分组，不再长成一排小胶囊工具。
- 右上区中的 `Language` 保留可见文字状态，但只作为 `轻系统状态`，不再作为一颗高声量独立 utility 按钮。
- `Transport workbench` 中真正承担区域标题职责的是 `column title / section title`。
- `Transport workbench` 中的 `kicker` 只允许作为轻辅助锚点，不再与标题共同构成双层强标题系统。
- `Transport workbench` 中的 `meta strip / meta pill` 统一按 `可读元信息` 处理，不按准控件处理。
- 像 `section-header` 这类当前全局混用的标题家族，后续必须拆成不同角色家族，不再继续靠区域覆盖勉强共用。

## 明确否决或受限方向

### 一、结构方向
- 不把“把更多面板压成 Adobe 式工具条”作为第一解法。
- 可以借 `工具带的占地效率`，但不借 `把复杂编辑图标化` 的工作模型。
- 不保留 `Inspector / Project` 作为右栏中的平级双 tab 主关系。
- 不保留 `bottom dock` 作为综合工具区或第二工作台。
- 不允许 overlay、popover、hint 继续承担长期悬挂的半工作区职责。
- 不采用 `纯自动` 的 `scope` 切换策略。
- 不采用 `纯手动` 的 `scope` 切换策略。
- 不采用 `每次新选择都重置` 的手动锁定生命周期。
- 不把左栏继续扩张成缩小版主工作栏。

### 二、Typography 与视觉语言方向
- 不把“换一套更有设计感的字体”当成字体问题的主解法。
- 不走 `双主字族` 对比系统。
- 不靠新 webfont 来制造个性。
- 不让英文和中文处于不同完成度。
- 不让轻策略感扩散到大量标签、pill、button 标题和控件文案中。
- 不保留大量 `kicker / eyebrow / mini-label / pill-label` 继续各说各话。
- 不让状态文字继续作为独立 typographic 等级存在。
- 不让表单标签、控件标签继续长得像小标题。
- 不用更多卡片感、浮起感来补“完成感”。
- 不扩成多强调色系统。

### 三、按钮语法与控件层级方向
- 不把 `icon-only` 当成默认按钮语言。
- 不允许 `icon-only` 扩散到设置入口、危险动作和项目级动作。
- 不允许一个可见工作面同时出现多个互相争抢注意力的 primary action。
- 不允许 `popover` 继续承担阅读、比较、填写、确认、项目级设置这类长停留职责。

## 讨论记录

### 2026-04-05 第一轮：结构骨架
- 先确认“地图优先”而不是“编辑台优先”。
- 用户提出希望借 Adobe 系列软件的工具条感，为地图让出更多视觉权重。
- 讨论后的结论不是直接把更多面板工具条化，而是只借工具带的轻量占地，不把复杂编辑内容图标化。
- 先后定下：
  - 右栏是唯一主工作栏
  - 左栏改为窄模式栏
  - 顶部条改为轻状态条
  - dock 默认收起
  - 浮层退回短时反馈
- 在右栏组织方式上，确认 `项目级` 和 `对象级` 内容不是互斥关系，但也不应被表达为平级双 tab，而应被表达为同一工作栏中的不同 `scope`。
- 在引导结构上，确认单靠左栏说明不够，因此采用“双层引导”：左栏解释模式，右栏解释当前任务。

### 2026-04-05 第二轮：Typography 与视觉语言
- 先确认这轮不先选具体字体名，而是先定字体系统应该有几种角色。
- 重新确认当前问题不是“没个性”，而是字体角色太多、层级太碎、状态文案与标题文案互相伪装。
- 确认产品不追求“很多种会说话的小标签”，而是追求“少数几种明确角色，各自只负责一件事”。
- 在主字体系上，确认：
  - 走单主字族
  - 主字族倾向人文 Sans
  - 双语平权是长期前提
  - 不引入新 webfont 作为第一步
- 在轻策略感的位置上，确认它只允许留在 `app 名 + 主壳标题` 这类极少数壳层锚点上。
- 在气质表达方式上，确认壳层锚点主要依靠比例与字重，而不是 overline、颜色和装饰。
- 在角色收口上，确认现有大量小标题 / 小标签角色要大幅并轨，后续不再允许它们继续并列扩散。
- 在数值语言上，确认数值 / 坐标 / 计量 / ID / 快捷键应正式独立成另一种语言。
- 在排版节奏上，确认整体走“中性偏紧凑”，既不展示页化，也不硬核压迫化。
- 在表面语言上，确认完成感来自平面层次与秩序，而不是更多卡片感和浮起感。
- 在颜色策略上，确认保留 `UI 冷色 + 地图暖色` 的双强调分工。
- 在状态文字上，确认它不是一级新的字体层级，而是语义，需要回归正文或元信息。

### 2026-04-05 第三轮：按钮语法与控件层级
- 先确认当前问题不是“按钮不够精致”，而是同一个职责被多种控件语法同时扮演。
- 先收住总方向：产品按钮系统改为 `文本优先`，不再追求大面积 icon-first。
- 用户明确接受 icon-only 例外，但只接受它留在高频工具、视口控制和少量壳层 utility 中。
- 讨论中明确收回 `语言切换` 的纯 icon-only 方向，确认它必须保留可见文字状态，因为它同时承担系统状态表达。
- 确认每个当前工作面默认只允许一个真正的 primary action，以压住视觉重心四散的问题。
- 确认唯一 primary action 默认跟随当前任务内容，放在任务段末尾，而不是固定悬浮到底部，也不是按区域漂移。
- 在 popover 权限上，确认它只负责局部、短停留、贴身的微操作，不再承担阅读、填写、比较和项目级设置。
- 用户额外提出高密度视觉操作区可以保留 popover；讨论后的限制是：允许，但必须严格限制数量，且不能长成第二工作区。
- 在切换态语言上，确认全产品默认只保留两种语法：
  - `major tabs`
  - `compact toggle / segmented`
- 确认工具选中态视为后一类的变体，不再允许 `tab / segmented / mode toggle / 伪 tab` 长期平行并存。
- 在危险动作位置上，确认 `Delete / Clear / Reset` 默认回到文字面板或明确详情区，不再进入顶条和主工具带。
- 在对象层动作上，确认列表默认不再长期暴露整排行内按钮，编辑、复制、删除等动作应收进单入口。
- 在列表到详情的关系上，确认点选对象 / 预设 / 条目时，默认行为应是直接进入右栏详情，而不是“先选中，再去找 Edit”。

### 2026-04-05 第四轮：壳层 utility 与任务动作分家
- 先确认当前核心问题不是“壳层不够像工作站”，而是壳层、支持工具和任务动作长期混线。

  - 支持工具和任务动作全部退回右栏或任务区
- 在 `Transport workbench` 的定位上，确认它属于 `全局 workspace 入口`，不是普通任务动作，也不是顶条状态项。
- 在壳层常驻项上，确认壳层只常驻环境与视口，不再长期常驻 `Reference / Export / Guide` 这类支持工具。
- 在支持工具落点上，确认 `Reference / Export / Guide` 的正式落点统一回到右栏 `项目 scope`。
- 在顶条权限上，确认顶部 `scenario context bar` 不再承载任务动作，只保留状态信息和一个固定的帮助 / 引导入口。
- 在顶条唯一固定动作的性质上，确认它属于 `全局帮助 / 引导`，而不是 workspace 入口，也不是当前模式快捷动作。
- 在 workspace 入口位置上，确认 `Transport` 从顶条移出，进入右上 utility 区。
- 在右上壳层密度上，确认生产态常驻密度按 `功能组` 计数，默认不超过 `3 组`。
- 在左右侧栏开合按钮的定位上，确认它们继续贴边独立存在，不并入右上 utility 区。
- 在 `Guide` 行为上，确认它只是壳层轻入口，点击后默认切到右栏支持区，而不是继续承担完整壳层弹层。
- 在实施节奏上，确认后续落地采用 `两波收口`：
  - 第一波做入口重归位并删除已确认重复入口
  - 第二波再做壳层与任务动作的视觉语法统一
- 在第一波范围上，确认第一波不仅要重归位，还要直接删除已确认重复的壳层入口，先把结构噪音压下去。

### 2026-04-05 第五轮：信息密度与留白节奏
- 先确认当前问题不是简单的“某些块太挤”，而是太多区域同时使用高声量紧凑节奏，导致视觉主次被压平。
- 先收住全局基调：产品整体的信息密度改为 `外松内紧`，不再让“紧”成为首屏和壳层默认态。

- 在章节内部顺序上，确认默认节奏是 `先列表，后详情`，先给用户扫描空间，再进入对象后展开详情与动作。
- 在帮助文案显性级别上，确认默认应 `隐一点`，只保留最必要的一句，更多解释按进入对象、错误、空状态或悬浮再触发。
- 在 `bottom dock` 的空闲态存在感上，确认它默认保持 `弱存在`，不再以高声量主工具台长期压在地图前方。
- 在左侧 rail 的气质上，确认它默认是 `安静入口脊柱`，负责入口和少量状态提示，而不是另一条常驻高密度小工具带。
- 在密度系统复杂度上，确认后续只保留 `松 / 中 / 紧` 三档，不再发明更细碎的中间档位。
- 在档位分配原则上，确认采用 `区域身份优先`，先看它属于地图、壳层、右栏还是局部专业块，再决定可以有多紧。
- 在 `紧` 的准入范围上，确认它只允许留给 `局部专业块`，例如对象列表、局部工具组、兵牌或图层微控件，不允许扩散到大章节和壳层本体。

  - 按目标身份归入 `松`
  - 状态文字归入 `元信息状态`
  - 默认 `主状态突出，其余弱化`
  - 最多只留一个固定轻动作
  - 状态项更接近轻文本分组，不再长成一排小胶囊工具

  - 整体归入 `中`

  - `Language` 并入视口语境，只作为轻系统状态
  - `workspace / Transport` 在这里保持低频入口感，而不是当前任务按钮感
- 在左侧 rail 上，确认：
  - 整体归入 `松`
  - 强标题归壳层锚点
  - `Scenario / Appearance` 只作为入口标签
  - 左栏里的 `section-header` 只作为入口摘要标签
  - 左栏动作只属于轻动作
- 在 `Transport workbench` 上，确认：
  - 它是合法的 `紧` 档局部专业工作区
  - 真正的区域标题只留给 `column title / section title`
  - `kicker` 只做轻辅助锚点
  - `meta strip / meta pill` 只做可读元信息
  - 真正最紧的区域落在 `section body / field / inspector rows`
- 在全局纪律上，确认：
  - 文字系统走 `一角一义`
  - 密度系统走 `区域先有默认档，再开局部例外`
  - `kicker / eyebrow / mini-label` 只允许壳层锚点与少量专业工作区保留例外
  - 像 `section-header` 这类当前全局混用家族，后续必须拆成不同角色家族
- 在右栏 `project scope` 上，确认其默认密度为 `标准中档`，不按对象细节区的高密语气处理。
- 在左栏 `Appearance` 上，确认其身份只应是入口层，不应继续长期维持深工作区结构。

### 2026-04-05 第七轮：`project scope` 结构秩序收口

- 确认 `Frontline` 在默认空闲状态下采用 `常驻次主区`，长期可见，但必须明确低于 `Project & Legend` 一档。
- 确认 `Utilities` 的身份是 `支持工具入口层`，不再被视为正式主章节。

- 确认 `Utilities` 内部入口语法走 `文字入口为主`，icon 只作为辅助，不再让支持工具长回小工具台。

- 确认 `Frontline` 和 `Utilities` 虽然当前都借用了 `summary.section-header` 结构壳，但后续视觉语法必须正式拆成两种角色，不能继续共用同一种标题话术。
- 将 `project scope` 的结构秩序压成一句话：
  - `Project` = 壳层锚点
  - `Project & Legend` = 主锚点
  - `Frontline` = 常驻次主区
  - `Utilities` = 尾部轻入口

### 2026-04-05 第八轮：5 级文字矩阵与 class 归位规则

- 确认 `Utilities` 未来不再算正式章节，只保留为 `支持入口头`。
- 确认 `Diagnostics` 未来不再算正式章节，只保留为 `附录头`。

  - 真正的错误、冲突、缺失、阻塞，最多只允许抬升到 `正文说明层`
  - 状态不允许越级抢到标题层

  - 只有在 `当前任务核心结果` 中，少数关键数值才允许明显抬升

  - `入口标题`
  - `正式章节标题`
  - `工具内容标题`

  - `入口标题` 更接近导航标签，不接近内容标题
  - `正式章节标题` 与主锚点必须 `清楚拉开`
  - `工具内容标题` 必须 `低于正式章节标题`
- 确认 `inspector-section-summary` 的未来角色只保留为 `章节标题壳`，不再给 `Utilities / Diagnostics` 这类非正式章节继续借用。

### 2026-04-05 第九轮：第一波 class 清理边界与拆分策略

### 2026-04-05 第十五轮：状态类元素归位表
- 这一轮专门解决“状态、说明、读数”长期混线的问题。
- 这张表的目的不是发明更多状态组件，而是把现有状态类元素压回三个已经定下来的层级里。
- 从这一轮开始，任何状态元素都要先回答：它是在交代事实、解释情况，还是读取数值。

#### 三类状态层级表

| 层级 | 它真正负责什么 | 典型内容 | 默认声量 | 明确限制 |
| --- | --- | --- | --- | --- |

| `正文说明` | 解释为什么现在是这样、下一步要做什么、为什么不可用 | 空状态提示、冲突说明、校验提示、帮助短句 | 中低 | 不伪装成标题，不装成 badge，不做行内装饰标签 |
| `信息读数` | 读取量化信息或结果值 | 数量、百分比、尺寸、层级、强度、计数 | 低到中 | 默认不是状态，不自带情绪色，不单独冒充章节锚点 |

| 空状态标题下那一行解释 | `正文说明` | 它解释为什么现在没有内容 | 留在空状态组，不再借标题类 |
| 冲突 / 缺失 / 阻塞说明 | `正文说明` | 它们在解释问题，而不是陈述客观枚举状态 | 可以抬升，但最多抬到说明层，不越级成标题 |
| 工具内轻帮助文 | `正文说明` | 它负责减轻理解成本 | 优先用一句短说明，不另发明问号体系 |

| 当前任务核心结果中的关键数值 | `信息读数` | 它仍然是读数，只是重要级别更高 | 允许局部抬升，但不能抢走章节标题职责 |

#### 混线警戒

- 只要一段文字里出现“为什么”“请先”“当前无法”“下一步”，它默认更接近 `正文说明`，不该伪装成 `chip`。
- 只要一个元素主要回答“现在是什么”，它默认更接近 `元信息状态`，不该被写成长说明句。
- 只要一个元素主要回答“数值是多少”，它默认就是 `信息读数`，不该额外伪装成状态 badge。
- 真正的错误、冲突、缺失、阻塞，最多抬升到 `正文说明`，不因为严重就自动升级成标题层。
- `chip / pill / badge` 以后只能承载短状态，不再承载解释、引导和长标签。

| 档位 | 它真正意味着什么 | 适合的区域 | 明确限制 |
| --- | --- | --- | --- |
| `松` | 以留白、呼吸、弱存在感为主 | 地图本体、壳层边框、轻状态条、安静入口脊柱 | 不能偷偷塞进大量小胶囊和工具项 |
| `中` | 以稳定阅读和编辑为主 | 右栏正式章节、项目区、对象区、右上视口组 | 不能顺手滑成全局紧凑工作台 |
| `紧` | 只给局部专业块和高频微控件 | 局部列表、局部工具组、局部微控件、专业工作区内核 | 不能扩散成整栏整页默认语气 |

#### 代表性区域归位表

| 区域 | 默认档位 | 为什么在这一档 | 允许的局部例外 |
| --- | --- | --- | --- |
| 地图主画布 | `松` | 它是主静区，必须给视觉重心和操作空间 | 局部 marker / micro overlay 可局部变紧 |
| `scenario context bar` | `松` | 它是轻状态条，不是主工具带 | 当前主状态项可以局部稍强，但不改整条档位 |
| 左 rail | `松` | 它是安静入口脊柱，不是缩小版工作栏 | 当前激活项可以局部加强 |
| 右上 `zoom / utility` 组 | `中` | 它是视口控制区，要清楚但不能抢主舞台 | 组内 icon 可以更紧，但整体仍是中档 |
| 贴边 panel toggles | `中` | 它们是视口空间控制点，需要可见但不吵 | 仅按钮本体紧，不带出整条工具带 |
| `Project scope` 整体 | `中` | 它是正式阅读和编辑区，不该按对象细节高压处理 | 内部局部列表和微控件可局部变紧 |
| `Inspector scope` 整体 | `中` | 它是对象详情主区，需要稳定阅读节奏 | 具体对象列表、颜色控件、局部微组可变紧 |
| `Project & Legend` | `中` | 它是主锚点区，必须稳，不宜挤 | 图例局部行项目可局部紧 |
| `Frontline` | `中` | 它是次主区，不该因“像工具”就整块收紧 | 内部局部控件组可以紧 |
| `Utilities` | `中` | 它是尾部支持区，不该像工具仓库那样挤爆 | 入口按钮组可以局部紧，但头部仍中 |
| `Diagnostics` | `中` | 它是附录区，不该用整块紧来抢戏 | 内部调试读数可局部紧 |
| `bottom dock` 外层 | `松` | 它应弱存在，不该长期压在地图前面 | 当前任务组内部可以局部紧 |
| `bottom dock` 当前任务组内部 | `紧` | 它是短工具带内核，服务于高频切换和微动作 | 只限当前组，不外溢到整条 dock |
| 对象列表 / 预设树 / 区域列表 | `紧` | 它们是典型局部专业块 | 只在列表内部紧，不带动整章变紧 |
| `Transport workbench` | `紧` | 它是合法的局部专业工作区 | 只在独立战场内成立，不反向污染主界面 |

#### 分档禁令

- 不能因为一个区域“内容多”就直接把它改成 `紧`。
- 不能因为一个局部控件“需要密一点”就把整章一起拉进 `紧`。
- `松 / 中 / 紧` 只能按区域身份定默认档，不能按临时视觉焦虑随时漂移。
- `Transport workbench` 的紧凑语气不得反向成为主界面默认参考。
- `bottom dock` 只能“外松内紧”，不能整条一起变成硬核控制台。

## 当前母约束
- 后续任何实现讨论都要先服从结构骨架，再谈风格细节。
- 后续任何视觉讨论都要先服从 5 级文字矩阵，再谈具体字体名和尺寸。
- 后续任何按钮讨论都要先服从“职责先于造型”，先判断它是导航、切换、动作、展开、说明还是危险操作，再决定长相。
- 后续任何 icon 使用都必须先经过“是否属于豁免名单”的判断，不能再用“这个也挺直观”作为扩张理由。
- 后续任何壳层讨论都要先判断它属于环境状态、视口控制、workspace 入口、支持工具还是任务动作，再决定是否有资格留在边缘壳层。
- 后续任何边缘入口讨论都要默认先问“它是否真的必须常驻壳层”，而不是默认把新功能抬上去。
- 后续任何密度与留白讨论都必须先服从 `区域身份优先 + 三档密度` 规则，再谈具体 padding、gap 和容器形式。
- 后续任何区域分档都必须先给出默认档，再说明局部例外，不能再让整个区域随状态整体漂移。
- 后续任何章节设计都要默认先问“它能否主要靠留白成立”，而不是先给它一层 card。
- 后续任何高密度块都必须先经过“它是否属于局部专业块”的准入判断，不能再把紧凑节奏随手扩散到整栏整区。
- 后续任何文字类讨论都必须先服从 `一角一义`，不能再接受“同名不同义，先混着用”。
- 后续任何标题类改动都要先判断它属于壳层锚点、当前任务标题、结构标签、入口摘要标签、元信息标签中的哪一种，再谈样式。
- 不能再通过新增角色、补充标签、增加浮层去修复“看起来不像完成品”的焦虑。
- 完成感必须来自更清楚的比例、节奏、数值秩序、平面层次，而不是来自更多 UI 话术。
