# 日本工业区试点探索归档

日期：2026-03-28

## 1. 一句话结论

如果日本这条线要求的是 `真实园区 / 工業用地`，而不是“工业规划分区”，那么国土数値情報 [L05 工業用地データ](https://nlftp.mlit.go.jp/ksj/gml/datalist/KsjTmplt-L05.html) 虽然较旧，但仍然是当前最贴题的主源；[A29 用途地域データ](https://nlftp.mlit.go.jp/ksj/gml/datalist/KsjTmplt-A29-v2_1.html) 只能作为比较和代理备选。

## 2. 研究边界

本轮固定边界如下：

- 研究对象是 `真实工业园区 / 工业用地`
- 首版几何是 `面`
- 不用规划分区直接替代真实工业区
- 不把工业地产、产业政策园区名单、开发区新闻名录混在一起

## 4. 为什么主源应该选 L05

L05 的关键不是“工业”两个字，而是它定义的对象就已经更接近产品目标。

页面说明里写得很清楚：

- 它覆盖 `工業団地`
- 也覆盖 `単独工場用地`
- 范围门槛是 `10ha 以上`
- 几何本身就是范围面

这和“工业区 / 工业用地”的产品语义是对齐的。

A29 虽然更新得更近，但它的本体是 `用途地域`，也就是城市规划管制意义上的用途分区。

这两者不能混为一谈。

## 5. 日本试点主源 / 后备源 / 排除项

### 5.1 主源

- 国土数値情報 `L05 工業用地データ`

### 5.2 后备增强

- 国土数値情報 `A29 用途地域データ`，仅用于比较“真实工业用地”和“规划工业分区”的差异
- 各地方工业园区公开名单，仅用于抽样核对园区名称和运营主体

### 5.3 当前排除

- 单纯的规划工业用途分区替代层
- 招商目录层
- 产业政策园区名单层
- 企业厂房 POI 点层

## 6. 与现有仓库架构的承接判断

这条线不适合先做点层。

原因：

- 工业区的核心价值在范围
- `临海 / 内陆`、园区名称、开发主体都更适合挂在面对象上
- 如果点化，用户看到的就不再是“园区/用地”，而是“园区中心点”

因此推荐承接方式固定为：

- `industrial_zones` 独立 deferred context layer pack
- polygon/context layer 渲染模式
- 可选的面样式分级和轻量标签

## 7. 当前最该提前写清的风险

- L05 很容易被嫌“旧”，但如果不接受“旧但最准”，就会滑向更近年的规划代理层。
- A29 的许可边界比 L05 更复杂，因为 2019 版还要求遵守各市区町村的单独使用条件。
- 如果后面想补更多工业园区名录，很容易把一个“范围层”做成“招商信息拼盘”。

## 8. 下一步建议

如果继续推进日本试点，这条线的最短路径是：

1. 先用 L05 抽样核东京湾、伊势湾、濑户内海、北九州几类工业密集区
2. 验证 `工業団地` 与 `単独工場用地` 两种对象是否都值得保留
3. 先做 `zone_type` 与 `coastal_inland` 两个最小产品字段
4. 把 A29 明确锁死为比较层，而不是后门替代主源

## 9. 试点判断

在这三条专题层里，`工业区` 是最适合优先进入产品试点的对象。

原因很直接：

- 产品语义最清楚
- 主源与目标最对题
- 面层表达能直接体现“真实园区 / 工业用地”的价值
- 不需要先解决复杂的网络问题，也不需要把“资源分布”和“现势经营状态”混在一起
