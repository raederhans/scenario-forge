# 日本铁路试点方法论归档

日期：2026-03-28

## 1. 目标

这份文档单独归档 `日本铁路试点` 的方法论，目标不是马上开工，而是把以后真正实现时需要坚持的口径先写死，避免需求继续漂移。

试点目标：

- 做一个 `半业务图层`
- 国家范围：`日本`
- 内容范围：`铁路线 + 主要车站`
- 数据策略：`官方源优先，OSM 补缺`
- 接入方式：`独立 deferred context layer pack`

## 2. 为什么选日本

日本是当前最有代表性的铁路试点国家，原因有四个：

1. 官方公开铁路数据成熟。
2. 铁路密度高，图层价值在产品里会非常直观。
3. 同时存在高铁、干线、支线、城市铁路，能把等级映射问题一次测出来。
4. 如果这套方法在日本能跑通，后续复制到加拿大、美国会更容易，而不是更难。

## 3. 首版范围

## 3.1 做什么

- 铁路线显示
- 按状态筛选
- 按等级筛选
- 点击查看线路属性
- 显示主要车站
- 点击查看车站属性
- 支持项目保存与恢复

## 3.2 不做什么

- 不做时间轴
- 不做全量车站
- 不做线路编辑
- 不做 routing
- 不做运行图、班次、客货流量
- 不做全国人工精修历史废线体系

## 4. 数据源策略

## 4.1 主源

主源使用日本国土数値情報的铁路数据。

参考：

- [产品规格 PDF](https://nlftp.mlit.go.jp/ksj/gml/product_spec/KS-PS-N02-v3_1.pdf)

主源承担的责任：

- 现役线路的基础几何
- 官方线路命名
- 官方站点命名
- 官方结构口径

## 4.2 补充源

补充源使用：

- [OpenStreetMap Railways](https://wiki.openstreetmap.org/wiki/Railways)
- [Geofabrik 日本提取](https://www.geofabrik.de/data/download.html)

补充源只承担以下责任：

- `disused`
- `abandoned`
- `construction`
- 官方源明显缺失的边缘要素

明确不做：

- 不把 OSM 现役线路全量覆盖到官方主源之上。
- 不把 OSM 当成和官方并列的双主源。

## 4.3 这样选的原因

- 官方源给我们法律边界更清晰的主网络。
- OSM 给我们更灵活的生命周期状态信息。
- 两者互补，而不是互相打架。

## 5. 内部统一 schema

## 5.1 线路 schema

首版内部字段建议固定为：

| 字段 | 含义 | 是否首版必需 |
|---|---|---|
| `id` | 稳定主键 | 是 |
| `name` | 线路名 | 是 |
| `source` | `official_jp` / `osm_patch` | 是 |
| `status` | `active/disused/abandoned/construction` | 是 |
| `class` | `high_speed/trunk/branch/service` | 是 |
| `operator` | 运营主体 | 否，能稳拿到再上 |
| `notes` | 去重或映射说明 | 否，内部可保留 |
| `geometry` | 线几何 | 是 |

## 5.2 站点 schema

主要车站只保留最少字段：

| 字段 | 含义 | 是否首版必需 |
|---|---|---|
| `id` | 稳定主键 | 是 |
| `name` | 站名 | 是 |
| `source` | `official_jp` / `osm_patch` | 是 |
| `city_key` | 所属城市键 | 是 |
| `importance` | 站点重要度 | 是 |
| `geometry` | 点几何 | 是 |

## 6. 等级与状态映射规则

## 6.1 状态

状态先统一成四档：

- `active`
- `disused`
- `abandoned`
- `construction`

映射原则：

- 官方主源默认归为 `active`
- OSM 中带明显废弃、停用、在建标签的要素进入对应状态
- 如果同一条线在官方中存在，但 OSM 只是局部维护状态差异，不允许整条覆盖，必须先做局部判定

## 6.2 等级

等级先统一成四档：

- `high_speed`
- `trunk`
- `branch`
- `service`

建议口径：

- `high_speed`
  - 新干线相关主通道
- `trunk`
  - 全国干线、主要区域骨架线
- `branch`
  - 一般支线、地方铁路主线
- `service`
  - 侧线、场站连接线、编组场或首版不强调显示的低价值段

注意：

- 这里的等级是 `产品显示等级`，不是日本铁路法律分类的完整翻版。
- 首版先求稳定、可筛选、可解释，不追求一次就完全学术严谨。

## 7. 主要车站筛选方法

当前已定策略是 `行政/城市权重优先`，因此不走“全量铁路枢纽算法优先”。

### 7.1 首版规则

优先纳入：

- 东京
- 都道府县厅所在城市核心站
- 全国主要大城市核心站
- 与 `high_speed` 线路直接相关的重要站

### 7.2 同城多站处理

一个城市如果有多个候选站，首版默认只保留一个主站，排序建议如下：

1. 新干线站优先
2. 城市主站优先
3. 跨线换乘能力更强者优先
4. 如果仍然冲突，再用人工白名单定主站

### 7.3 为什么不先上全站

- 全站会把问题从“图层方法论”升级成“全国站点产品设计”。
- 样式、缩放、筛选、点击优先级都会明显复杂。
- 首版只需要证明铁路层有价值，不需要一次把站点系统做完。

## 8. 官方源与 OSM 的补齐规则

这是整个试点里最关键的一条。

### 8.1 主原则

- 官方主源决定 `active` 主网络。
- OSM 只补官方没有、但产品上确实需要的部分。

### 8.2 补齐顺序

1. 先导入官方线路。
2. 再导入 OSM 日本铁路提取。
3. 只筛选以下 OSM 要素进入候选：
   - `disused`
   - `abandoned`
   - `construction`
   - 官方明显缺失的局部现役线
4. 做空间去重与名称比对：
   - 与官方主源高度重合的现役线直接丢弃
   - 只保留真正新增的生命周期补充段

### 8.3 去重策略

首版不要做复杂启发式。

建议只用严格规则：

- 空间重合度阈值
- 名称匹配
- 状态优先级
- 人工白名单

如果严格规则不能决定，就先不并入，不做模糊补丁。

## 9. 构建方法

## 9.1 后端

后端只做两类产物：

1. `railways` 独立 TopoJSON pack
2. `rail_stations_major` 独立 GeoJSON 或 TopoJSON pack

后端处理步骤：

1. 读取日本官方铁路线路
2. 读取日本官方站点
3. 读取 OSM 日本铁路提取
4. 转成统一 CRS
5. 裁剪到地图范围
6. 规范字段
7. 统一状态与等级
8. 做严格去重
9. 为线路和站点生成稳定 ID
10. 输出独立延迟加载包

## 9.2 前端

前端分成两层：

- `railways`
- `railStations`

前端行为：

- 默认不加载
- 用户打开开关后才加载
- 缩放越低，显示越少
- 低缩放只显示 `high_speed + trunk`
- 中缩放增加 `branch`
- 高缩放再放 `service`、`disused`、`abandoned`

## 10. 交互规则

## 10.1 线路点击信息

首版点击线路至少显示：

- 名称
- 状态
- 等级
- 来源

## 10.2 站点点击信息

首版点击站点至少显示：

- 名称
- 所属城市
- 重要度
- 来源

## 10.3 筛选维度

首版只开放两类筛选：

- 按状态
- 按等级

不要一开始开放运营商、电气化、复线、速度等高级筛选。

## 11. 验收标准

日本试点通过的最低标准：

1. 图层能独立懒加载。
2. 首屏性能不因为铁路图层而明显变差。
3. 低缩放下不会糊成一片。
4. 线路状态筛选可用。
5. 线路等级筛选可用。
6. 主要车站显示可读。
7. 项目导入导出后，图层状态和样式能恢复。

## 12. 未来扩展顺序

日本跑通后，建议按这个顺序扩：

1. 加拿大铁路
2. 美国铁路
3. 欧盟单国 INSPIRE 样板
4. 欧洲多国铁路
5. 公路/高速图层
6. 港口、机场、航道等其他基础设施

## 13. 一句话方法论

日本铁路试点的核心方法论是：

`用官方全国铁路数据定义现役主网络，用 OSM 只补生命周期状态和缺口，用独立 deferred pack 接进现有 context layer 架构；首版只证明“状态 + 等级 + 主要车站”这一套半业务图层能否成立，不提前做成复杂时序系统。`
