# 中国机场专题研究归档

日期：2026-03-28

## 1. 一句话结论

中国机场这条线最适合先试点，当前最稳的写法是 `官方名录/统计 + 点位补充`：中国大陆的官方运输机场公报和吞吐量口径很强，台湾则同时具备较强的政府开放机场点位与营运资料；因此中国机场完全可以先做成 `主要公共/民航机场` 点层。

## 2. 研究边界

- 只研究 `设施本体`
- 只做 `点图层优先`
- 只优先收 `主要公共/民航机场`
- 不研究航线
- 不研究机场范围面
- 不研究军用专题设施
- 不研究直升机场专题

## 4. 主源 / 后备源 / 排除项

### 4.1 主源

- 大陆对象范围与重要度：`全国民用运输机场生产统计公报`
- 台湾官方点位与营运补强：`機場(離島)`、`桃園機場基本資料`、`各機場、港口入出境人數統計資料`

### 4.2 后备源

- `OSM / Geofabrik`：补大陆点位和两侧缺失代码或别名
- 机场运营方公开页面：补个别枢纽机场属性

### 4.3 排除项

- 航线网络
- 军用机场专题
- 直升机场专题
- 把大陆数据目录型页面误写成可直接下载的机场点层

## 5. 大陆与台湾差异

### 5.1 大陆

- 官方运输机场统计、公报和等级口径非常强。
- 公开可直接批量拿到的机场点层没有台湾顺。
- 因此大陆部分应写成 `官方名录/统计 + 点位补充`。

### 5.2 台湾

- 官方开放平台直接给到了机场点位类数据和营运数据。
- 这使台湾部分比大陆更接近 `官方点位主源可用`。
- 但在中国整体口径下，仍不应伪装成已经有一套统一的中国官方机场点层。

## 6. 与现有仓库架构的承接判断

- 最适合直接做成 `airports` 点层
- 前端复用现有 `cityPoints` 风格链路即可
- 构建期按 `mainland / Taiwan` 分别筛选主要机场，再合并成统一输出

## 7. 与日本最明显的不同

- 日本机场更像“官方专题点包先在”。
- 中国机场更像“大陆统计强、台湾点位强，两边拼起来才能形成一套实用研究结论”。
- 这也解释了为什么中国机场适合先试点，但仍不应过度承诺单一官方主包。

## 8. 风险与下一步建议

1. 最大风险是把大陆的统计公报误写成点位主源。
2. 第二个风险是把台湾离岛机场点位误写成覆盖全部台湾主要机场的完整主表。
3. 第三个风险是因为点位不够完美就拖延落盘；这轮不该这样，应先用官方名录和统计把对象收住。
4. 如果后续真要做中国机场首版，建议先只收主要公共/民航机场，不做全航空设施。
