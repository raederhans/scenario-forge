# 德国交通专题研究总览

日期：2026-03-28

## 1. 一句话结论

德国这四条交通线里，`公路` 最适合先试点，`机场` 和 `港口` 也能较快落地，`铁路` 则可以做成上下文层，但必须明确区分“公开官方骨架可用”和“运营级官方开放主源缺口仍然存在”。

## 2. 四条线总体成熟度判断

| 线别 | 成熟度判断 | 现在能不能直接落地 | 核心原因 |
|---|---|---|---|
| 公路 | 高 | 可以 | `BKG DLM250` 已经能作为全国官方开放主干道路几何主源，足够支撑 `motorway / trunk / primary` 上下文层 |
| 机场 | 中高 | 可以 | `BKG POI-Open` 可直接提供全国官方开放点位，主要问题是还要用 `ADV` 之类来源把“主要民航机场”筛出来 |
| 港口 | 中高 | 可以 | `BKG POI-Open` 可提供全国官方港口点位，但“主要商港/关键节点”仍需 `DMZ` 等准官方来源补强语义 |
| 铁路 | 中 | 可以，但要写清边界 | 公开官方骨架和客运站点都能拿到，但更细的 `StreckeDB` 不对公众开放，所以铁路只能稳做上下文层，不能冒充运营级官方专题层 |

当前排序：`公路 > 机场 > 港口 > 铁路`

## 3. 最适合先试点的一条线

最适合先试点的是 `公路`。

原因很直接：

1. 官方全国开放主源最清晰。
2. 任务边界已经被压到 `motorway / trunk / primary`，很适合 `DLM250` 这类官方基础地理产品。
3. 架构接法几乎可以直接复用日本公路方案，不需要先解决复杂设施筛选问题。
4. 许可风险也最小，不需要先依赖受限或模糊来源。

## 4. 与日本最明显的不同

和日本相比，德国交通专题最明显的不同是：德国更依赖“官方基础地理开放层 + 第二层业务筛选/补强”，而不是“一套官方专题包直接贴近业务对象”。

- 公路：德国比日本更适合直接用官方全国开放骨架起步。
- 机场和港口：德国都有官方点位主源，但业务语义要再筛一次。
- 铁路：德国不像日本那样容易找到既公开又贴近业务对象的专题铁路包，公开层更偏骨架，细层受限。

## 5. 哪些地方必须降级到非官方但可信公开源

- 铁路：如果要补细分类、非 DB 线、线名细节或局部缺口，必须降级到 `OSM / Geofabrik`。
- 机场：如果要统一补 `IATA/ICAO`、运营方等富属性，全国官方开放主源不够时，需要降级到 `OSM` 或机场方公开资料。
- 港口：如果 `POI-Open` 个别命名或点位不理想，可降级到 `OSM` 补缺；但“主要港口”本身优先用 `DMZ` 这类准官方公开来源收敛，而不是直接靠协作源决定名单。
- 公路：首版不必降级；只有在明确需要更细编号和分类时，才补 `OSM`。

## 6. 与现有仓库架构的承接判断

德国四条线都能复用既有方法论，但落点不同：

- `road`：线层，直接复用日本公路的 `deferred context pack`
- `rail`：线层加点层，线用官方骨架，点用官方站点，必要时再补 `OSM`
- `airports`：点层，直接复用现有 `city points` 风格的设施点层链路
- `ports`：点层，和机场一样按点层承接

也就是说，德国不需要新发明第四套架构，重点只是把“几何主源”和“业务筛选层”分开。

## 7. 风险与下一步建议

### 7.1 风险

1. 德国很多官方开放交通数据更偏基础地理表达，不应被误写成专题业务层。
2. 机场和港口如果不做筛选，很容易退化成通用 POI 集合。
3. 铁路最容易在文档里说得过满，尤其是把公开骨架误包装成完整官方专题铁路网。

### 7.2 下一步建议

1. 先做德国 `road`，因为最稳、最容易交付。
2. 第二梯队做 `airports` 和 `ports`，都以官方点位主源起步，再补业务筛选层。
3. `rail` 放在第四位，但不是不能做，而是要明确写成“上下文铁路层”，不要偷换目标。
