# 工业矿产、能源设施与工业区图层方法论归档

日期：2026-03-28

## 1. 目标

这份文档用于归档 `工业矿产`、`能源设施`、`工业区` 三条专题层的方法论，作为日本试点和后续跨国家复制时的统一口径。

当前方法论只面向：

- 地图中的 `半业务专题图层`
- 资源点、设施点、园区面三种基础表达
- 当前版本数据
- 可延迟加载的 context layer

当前方法论明确不面向：

- 现役矿山经营状态追踪
- 输油、输气、输电等线状网络
- 产业链分析、产能估算、物流流向
- 工业规划审批与动态时序回溯

## 2. 产品定位

这三类对象不能再被当成一个笼统的“工业经济大层”。

首版必须明确拆成三条专题层：

- `mineral_resources`：资源点层
- `energy_facilities`：设施点层
- `industrial_zones`：园区面层

它们可以共用同一套研究模板，但不能共用同一种产品语义。

## 3. 核心方法论

### 3.1 先定产品语义，再选数据源

这三条线最容易出错的地方，不是“找不到数据”，而是先拿到数据再反推产品定义。

本轮固定口径如下：

- `工业矿产` 指 `矿床/资源分布`
- `能源设施` 指 `点状设施`
- `工业区` 指 `真实园区/工業用地`

这意味着：

- 矿产层不是现役矿山层
- 能源层不是能源网络层
- 工业区不是规划分区代理层

### 3.2 点层和面层必须分家

`工业矿产` 与 `能源设施` 更接近现有仓库里的点图层体系，首版应该优先复用 `cityPoints` 的承接链路。

`工业区` 是范围对象，不应该被硬塞进点设施模型。它更接近已有的 polygon/context layer 方式。

因此首版几何策略固定为：

- `mineral_resources`：点
- `energy_facilities`：点
- `industrial_zones`：面

### 3.3 口径取舍必须显式写清

这三条线的取舍逻辑不同：

- `工业矿产` 优先 `资源分布优先`
- `能源设施` 优先 `全国统一口径`
- `工业区` 优先 `语义准确`

也就是说：

- 矿产层允许“不反映当前在产状态”
- 能源层允许“全国主源较旧，但口径统一”
- 工业区允许“主源较旧，但比规划代理更贴题”

### 3.4 不用兜底代理偷换概念

本轮不接受下面这些偷换：

- 用“现役矿山新闻名录”代替矿床分布
- 用“能源企业介绍页”代替全国统一设施库
- 用“用途地域中的工业类分区”直接代替真实工业用地

如果主源不够新，就明确写“旧但最准”。
如果全国统一源还不够完整，就明确写“只能做锚点，不承诺全能源设施全覆盖”。

### 3.5 统一研究模板

每条线都统一输出五段内容：

1. 一句话结论
2. 数据源判断表
3. 日本试点主源 / 后备源 / 排除项
4. 与现有仓库架构的承接判断
5. 风险与下一步建议

## 4. 推荐 schema

### 4.1 工业矿产

| 字段 | 含义 |
|---|---|
| `id` | 稳定主键 |
| `name` | 点位名称或矿山名 |
| `resource_type` | 矿种 |
| `resource_class` | 资源大类或成矿类别 |
| `work_status` | 可选，工作状态或图例状态 |
| `source` | 数据来源 |
| `geometry` | 点几何 |

### 4.2 能源设施

| 字段 | 含义 |
|---|---|
| `id` | 稳定主键 |
| `name` | 设施名称 |
| `facility_subtype` | 发电厂 / 炼油厂 / LNG 基地 / 变电站等 |
| `operator` | 运营主体 |
| `status` | 可选，运行 / 工事中 / 退役等 |
| `source` | 数据来源 |
| `geometry` | 点几何 |

### 4.3 工业区

| 字段 | 含义 |
|---|---|
| `id` | 稳定主键 |
| `name` | 园区或用地名称 |
| `zone_type` | 工业团地 / 单独工厂用地等 |
| `coastal_inland` | 临海 / 内陆 |
| `operator` | 可选，开发主体或管理主体 |
| `source` | 数据来源 |
| `geometry` | 面几何 |

## 5. 日本试点锚点

### 5.1 工业矿产

- 主锚点：GSJ [国内の鉱床・鉱徴地に関する位置データ集（第2版）](https://www.gsj.jp/publications/pub/prompt-rep/index.html)
- 主语义：资源分布
- 首版表达：点

### 5.2 能源设施

- 主锚点：国土数値情報 [P03 発電所データ](https://nlftp.mlit.go.jp/ksj/gml/datalist/KsjTmplt-P03-v4_0.html)
- 主语义：全国统一的点状能源设施锚点
- 首版表达：点

### 5.3 工业区

- 主锚点：国土数値情報 [L05 工業用地データ](https://nlftp.mlit.go.jp/ksj/gml/datalist/KsjTmplt-L05.html)
- 比较代理：国土数値情報 [A29 用途地域データ](https://nlftp.mlit.go.jp/ksj/gml/datalist/KsjTmplt-A29-v2_1.html)
- 主语义：真实工业园区 / 工业用地
- 首版表达：面

## 6. 与现有仓库的承接判断

### 6.1 点层

`工业矿产` 与 `能源设施` 可以优先复用现有 `cityPoints` 的承接链路，主要包括：

- `js/core/state.js`
- `js/core/map_renderer.js`
- `js/main.js`
- `js/core/file_manager.js`
- `js/ui/toolbar.js`
- `js/ui/sidebar.js`

可优先复用的能力包括：

- 点位渲染
- 标签密度控制
- 图层开关
- 项目持久化

### 6.2 面层

`工业区` 不应复用点层逻辑，而应沿用已有 polygon/context layer 的承接方式。

原因很简单：

- 工业区的值在于范围而不是中心点
- 点化会损失“真实园区/工業用地”的核心语义
- 后续如果要加面积、临海/内陆、开发主体等属性，面层承接更稳

## 7. 主要风险

- `工业矿产` 的最大风险不是几何，而是用户误把它理解成“现役矿山层”。
- `能源设施` 的最大风险是主源容易只覆盖发电设施，导致“能源设施”这个名字过宽。
- `工业区` 的最大风险是主源较旧，后续很容易有人想用更近年的规划分区来替换。
- 三条线如果混成一个大层，前端筛选、样式、状态和解释成本都会立刻变差。

## 8. 推荐推进顺序

如果三条线都要继续推进，日本试点下推荐顺序如下：

1. `工业区`
2. `工业矿产`
3. `能源设施`

原因是：

- `工业区` 的主源语义最稳定，最接近产品问题本身
- `工业矿产` 的主源权威性强，且点层承接简单
- `能源设施` 虽然点层也简单，但“能源设施”的边界天然比“发电所”更宽，最需要提前设边界

## 9. 一句话结论

这三类对象最稳的做法不是做成一个“大工业层”，而是：

`把资源点、设施点、园区面三条线分开建模，分别用最贴题的官方锚点跑通日本试点，再决定谁值得先进入产品。`
