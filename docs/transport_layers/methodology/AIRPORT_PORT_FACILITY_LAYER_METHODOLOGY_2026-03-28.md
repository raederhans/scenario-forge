# 机场与港口设施图层方法论归档

日期：2026-03-28

## 1. 目标

这份文档用于归档 `机场` 与 `港口` 设施图层的方法论，作为后续不同国家试点时的统一口径。

当前方法论只面向：

- 地图中的 `半业务设施图层`
- 设施本体
- 当前版本数据
- 可延迟加载的 context layer

当前方法论明确不面向：

- 航线
- 航路
- 货运流量
- 时序回溯
- 机场范围面与港口边界线的完整表达

## 2. 产品定位

机场和港口首版不是运输网络层，而是 `关键设施节点层`。

首版必须满足：

- 可单独开关
- 可点击查看基础属性
- 可按重要度筛选
- 可按类别筛选
- 可随项目导出和恢复

## 3. 核心方法论

### 3.1 先把设施本体和运输网络拆开

机场和港口和公路、铁路不一样。

这两类对象天然会同时引出两套东西：

- 设施本体
- 运输网络

比如：

- 机场 vs 航线
- 港口 vs 航路/海上流通线

首版必须先把设施本体单独做成稳定图层，不把网络层一起并进来。

### 3.2 点图层优先

机场和港口首版统一按 `点图层优先`。

原因：

- 更适合复用现有仓库里的点图层渲染链
- 更适合全国尺度展示
- 更容易控制密度
- 不会因为范围面或边界线把问题提前复杂化

机场范围面、港口区域界、港域界都属于后续增强，不进首版主链。

### 3.3 主要节点优先，不追求官方全量名录

设施类图层一旦全量收录，很容易退化成密集名录层。

因此首版默认采用：

- 机场：以 `民航/主要公共机场` 为主
- 港口：以 `国际战略港湾`、`国际拠点港湾`、`重要港湾` 为主

首版不是为了证明“我们能收齐多少设施”，而是为了证明“设施图层是否真的有产品价值”。

### 3.4 数据策略优先级

设施类图层优先采用：

1. 官方几何主源
2. 官方行政清单核对
3. 后备增强源

和公路不同，机场和港口首版不需要先默认走 `OSM 主几何`。

如果官方公开源已经能提供：

- 全国覆盖
- 稳定点位
- 足够的分类与名称

那首版优先走官方主源更稳。

### 3.5 机场和港口必须从第一天就分开建模

虽然两者都属于设施点图层，但它们不是同一类对象。

机场更像：

- 类型相对稳定
- 全国数据更整齐
- 点位和属性口径更统一

港口更像：

- 分类口径更碎
- 商港和渔港天然分叉
- 同时有点、区域界、港域界等多种几何
- 数据更新更容易滞后

所以首版必须共享一层通用 schema，但仍然保留各自的专属字段。

## 4. 推荐架构

### 4.1 接入方式

首版统一推荐：

- `独立 deferred context layer pack`
- `airports` 点图层
- `ports` 点图层

不建议：

- 塞进主拓扑
- 和线网图层混编
- 把机场、港口、航线、航路打成一个大层

### 4.2 优先复用现有点图层链路

当前仓库最接近的承接模式不是 `rivers`，而是 `cityPoints`。

可直接复用的骨架主要在：

- `js/core/state.js`
- `js/core/map_renderer.js`
- `js/main.js`
- `js/core/file_manager.js`
- `js/ui/toolbar.js`
- `js/ui/sidebar.js`

这意味着设施层首版可以优先复用：

- 点位样式
- 标签密度
- 开关状态
- 项目持久化

而不是另外发明一套散乱的点设施系统。

### 4.3 推荐的内部统一 schema

共享字段建议固定为：

| 字段 | 含义 |
|---|---|
| `id` | 稳定主键 |
| `name` | 设施名称 |
| `facility_type` | `airport` / `port` |
| `category` | 内部产品分类 |
| `importance` | 设施重要度 |
| `source` | 数据来源 |
| `geometry` | 点几何 |

机场补充字段建议：

| 字段 | 含义 |
|---|---|
| `airport_type` | 机场类别 |
| `operator` | 管理或运营主体 |
| `iata` | 可选 |
| `icao` | 可选 |

港口补充字段建议：

| 字段 | 含义 |
|---|---|
| `port_class` | 港口类别 |
| `manager` | 管理者 |

## 5. 首版验收标准

机场和港口设施层首版通过的最低标准：

1. 图层能独立懒加载
2. 点位密度在全国尺度下可读
3. 点击属性能稳定显示
4. 类别与重要度筛选可预测
5. 项目导入导出后状态可恢复
6. 不影响现有 context layer 的加载性能

## 6. 下一批高价值候选

如果机场与港口这一类设施图层跑通，下一批最值得继续收口的不是更复杂的航路，而是专题性更强的资源和设施层：

- `矿产/资源`
- `能源设施`
- `工业区 / 工業用地`

这些对象的共同特点是：

- 主题边界更清楚
- 图层目的更明确
- 点/面表达更容易建立产品价值

## 7. 一句话结论

机场和港口设施图层的正确起点不是“把所有运输信息都并进来”，而是：

`先把关键设施本体做成稳定、可读、可保存的点图层，再在清楚拆分设施层与网络层的前提下逐步扩展。`
