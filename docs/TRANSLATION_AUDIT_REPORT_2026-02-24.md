# 地图填色项目翻译系统审计报告（2026-02-24）

## 1. 任务目标
本次仅进行诊断，不改动业务代码。目标是回答以下问题：
1. 当前翻译脚本的真实状态如何。
2. UI 汉化不全的原因是什么。
3. 地块（geo）为什么只是打标签而没有实际翻译。
4. 后续如何改造，确保新增 element（UI、地块、说明文本等）都能进入翻译流程。

## 3. 当前翻译链路（现状）

### 3.1 数据来源
- 前端语言数据由 `data/locales.json` 提供。
- 启动时 `js/core/data_loader.js` 加载 `data/locales.json`，`js/main.js` 存入 `state.locales`。

### 3.2 UI 翻译机制
- UI 通过 `js/ui/i18n.js` 中的 `t(key, type)` 查 `state.locales[type][key]`。
- 当前语言为 `zh` 时取 `zh`；缺失时直接回退到 key（通常就是英文原文）。
- 多个 UI 模块（toolbar、sidebar 等）都依赖 `t()`。

### 3.3 地块（geo）翻译机制
- tooltip 和列表中的地块名称来源于 `feature.properties.name / NAME`。
- 这些名称同样通过 `t(name, "geo")` 查询 `locales.geo`。

### 3.4 词库生成机制
- `tools/translate_manager.py` 会：
  1. 从 `data/europe_topology.json` 抽取 geo 名称。
  2. 合并 `tools/geo_seeds.py` 的少量种子词条。
  3. 用 `MANUAL_UI_DICT` 生成/更新部分 `ui` 条目。
- 对未命中的 geo，脚本写入占位值：`"[TODO] <name>"`。

## 4. 关键问题与根因

### 4.1 UI 汉化不全的直接原因
- `data/locales.json` 的 `ui` 区块覆盖面不足（目前到 `Width` 一带），后续新功能新增的 UI 字符串并没有自动进入词库。
- 运行时对缺失 key 的策略是“回退英文”，因此表象是“部分汉化 + 大量英文残留”。

### 4.2 地块只有标签、没有真实翻译的直接原因
- `geo` 词库大量条目是 `"zh": "[TODO] xxx"`，说明只是被收集并标注待翻译，没有落地中文译文。
- `tools/geo_seeds.py` 覆盖范围有限，仅能翻译少量已预置名称。

### 4.3 系统性原因（为什么会退化成“屎山”感）
- 翻译来源分散：有手工字典、有脚本抽取、有运行时硬编码 key。
- 没有“新增字符串必须入库”的自动化守门（lint/CI/构建检查）。
- 缺少“翻译覆盖率”可观测性（无法快速看到哪些 key 在生产/开发中未翻译）。
- Geo 使用“显示名字符串”作为 key，天然容易出现大小写、别名、空格、拼写差异导致 miss。

## 5. 为什么新增 element 无法自动被翻译
现状缺少闭环：
1. 新增 element 文案出现于代码。
2. 没有统一抽取器把新 key 收集进词库。
3. 词库没有自动补全（机翻/术语库/人工校对流程）。
4. 构建阶段也不会因缺失翻译而失败。

结果就是：功能迭代越多，未翻译字符串越多，且只在界面上被动暴露。

## 6. 改进方向（不改业务行为前提下的架构建议）

### 6.1 建立单一事实源（Single Source of Truth）
- 统一用稳定 key，而不是英文原文做 key。
- 建议命名空间：
  - `ui.*`（界面）
  - `geo.*`（地块）
  - `hint.*` / `desc.*`（说明、帮助文案）
- 目标是“代码中只出现 key，不直接出现可见文案”。

### 6.2 新增自动抽取与同步脚本
新增一个“提取 + 同步”流程（可并入 `translate_manager.py` 或拆分脚本）：
1. 扫描 `js/`, `index.html`, `map_builder` 中可见字符串或 `t()` key。
2. 与 `data/locales.json` 对比，输出新增/缺失 key 清单。
3. 自动写入词库骨架（`en` 默认值 + `zh` 空值或机翻候选值）。

### 6.3 建立质量闸门（CI/本地 pre-commit）
- 新增检查：
  - 禁止出现未登记可见字符串（白名单除外）。
  - 禁止 `zh` 为 `[TODO]` 的 key 进入发布分支（或按模块设阈值）。
  - 检查重复 key、冲突 key、无引用 key。

### 6.4 Geo 翻译专项治理
- 不再直接用地名字符串当 key，改为稳定 ID（如 topo feature id / 自定义 canonical id）。
- 引入“别名归一化层”（canonicalization），先映射再翻译。
- 把 `geo_seeds.py` 升级为可扩展术语库（按国家/地区分文件，便于维护）。

### 6.5 运行时可观测性
- 在开发模式下统计 missing key 次数与来源页面。
- 提供“导出 missing keys”按钮或日志文件，回灌到翻译脚本。

## 7. 建议落地顺序（最小风险）
1. 第 1 阶段（低风险）
   - 增加 missing-key 扫描与报告，不改渲染逻辑。
2. 第 2 阶段（中风险）
   - 自动同步 locales 骨架，补齐新增 UI key 流程。
3. 第 3 阶段（中风险）
   - Geo key 从显示名迁移到稳定 ID，并做别名归一化。
4. 第 4 阶段（可选）
   - 接入机翻候选 + 人工审核工作流，提高翻译吞吐。

## 8. 结论
当前系统并非“翻译功能失效”，而是“翻译链路不闭环”：
- UI 端：新增 key 未持续入库，导致汉化覆盖率持续下降。
- Geo 端：大量条目停留在 `[TODO]` 占位，只有标签没有译文。

要实现“新增 element 自动可翻译”，核心不是继续堆手工词典，而是建立：
- 统一 key 规范
- 自动抽取/同步
- CI 覆盖率闸门
- Geo 稳定 ID 与术语库机制

以上改造可以分阶段实施，先提升可观测性和增量同步，再做结构迁移。
