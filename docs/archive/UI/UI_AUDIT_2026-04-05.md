## 主题一：首屏视觉重心不稳，地图没有真正成为主角

### 1.1 首屏同时存在四套并行控制平面
- 左侧 Workspace 树、右侧 Inspector、地图上方 scenario context bar、底部 dock 同时压在首屏里。
- 从浏览器实景看，地图虽然面积最大，但注意力会被四周的白色容器持续切碎；用户第一眼并不会自然聚焦到地图本体。
- 证据：
  - 截图：`.runtime/browser/manual-audit/full-home-steady.png`
  - 结构：`index.html:52`、`index.html:1233`、`index.html:1312`、`index.html:1494`

### 1.2 信息密度不是整体高，而是分布失衡
- 左栏几乎只剩 4 个一级入口，右栏和底 dock 却承担了大部分实际信息与动作。
- 结果不是“极简”，而是“左边空、右边挤、中间再被上下条带夹住”，视觉节奏很不均匀。
- 证据：
  - 截图：`.runtime/browser/manual-audit/left-sidebar-steady.png`
  - 截图：`.runtime/browser/manual-audit/right-sidebar-steady.png`
  - 截图：`.runtime/browser/manual-audit/bottom-dock-steady.png`
  - 结构：`css/style.css:250-276`、`css/style.css:1472-1487`、`css/style.css:4982-5142`

### 1.3 启动态先把“工作区气质”打断了
- 用户进入页面时，先看到的是一个全屏 boot overlay 和被降权的工作壳，不是地图工作区本身。
- 这会改变首屏的心理预期：它更像在等待一个系统，而不是直接进入地图编辑。
- 证据：
  - 截图：`.runtime/browser/manual-audit/full-home.png`
  - 结构：`index.html:2077-2103`
  - 样式：`css/style.css:101-123`、`css/style.css:140-179`

### 1.4 benchmark 对照
- ArcGIS Pro 明确把主界面拆成 `ribbon`、`views`、`panes` 三种不同层级，地图视图不会和所有操作层挤在同一层表达里。
- QGIS 直接把 `Toggle Panel Visibility` 和 `Toggle Map Only` 作为标准能力，等于官方承认“面板太多时必须让地图重新成为视觉中心”。
- 相关来源：
  - ArcGIS Pro: https://pro.arcgis.com/en/pro-app/3.4/get-started/introducing-arcgis-pro.htm
  - ArcGIS Pro pane sets: https://pro.arcgis.com/en/pro-app/3.5/get-started/pane-sets.htm
  - QGIS GUI: https://docs.qgis.org/3.10/en/docs/user_manual/introduction/qgis_gui.html

## 主题二：信息架构和交互层级套叠过深，顺序感弱

### 2.1 左侧栏不是单层导航，而是连续套娃
- `Scenario` 是一个折叠层。
- `Appearance` 先是总折叠，再是 tablist，再是 panel，里面还嵌 `Bathymetry Debug` 这样的二次折叠。
- `Special Zone Editor`、`Color Library` 也继续追加新的一级块。
- 用户会持续面对“先展开、再切 tab、再进 panel、再开 advanced”的层级跳跃。
- 证据：
  - `index.html:63-80`
  - `index.html:86-220`
  - `index.html:290-848`
  - `index.html:1114-1207`

### 2.2 右侧 Inspector 混用了页签导航和手风琴导航
- 顶部已经有 `Inspector / Project` 两个 tab。
- 每个 tab 内容内部又继续使用多段 `details`。
- 每段 `details` 里还塞 search、detail card、legend、hint、action button。
- 同一个边栏里同时存在“先切页签”和“再折叠卡片”的双导航语法，用户需要自己理解哪层才是主层。
- 证据：
  - `index.html:1496-1527`
  - `index.html:1542-1833`
  - `index.html:1773-1912`
  - 截图：`.runtime/browser/manual-audit/right-sidebar-steady.png`

### 2.3 首页临时层体系没有收敛
- 首页可见或可触发的临时层包括：
  - `scenarioGuidePopover`
  - `dockEditPopover`
  - `dockQuickFillRow`
  - `dockReferencePopover`
  - `dockExportPopover`
  - tooltip
  - `transportWorkbenchOverlay`
- 这些层的形态、尺寸、入口方式、语义都不同，但挂载在同一主壳里，导致“这到底是轻量提示、局部编辑，还是进入另一工作区”很难一眼分辨。
- 证据：
  - `index.html:1263`
  - `index.html:1379`
  - `index.html:1439`
  - `index.html:1863`
  - `index.html:1897`
  - `index.html:1924-2054`

### 2.4 URL 几乎没有承担界面状态层级
- 当前代码里只明显使用了 query 参数读取 startup / render profile。
- 没有看到首页 tabs、expanded panels、dock 折叠状态、左右面板状态写回 URL 的主线。
- 这意味着界面状态更像临时内存状态，不像一个可被深链接、可恢复、可共享的工作站布局。
- 证据：
  - `js/main.js:524-525`
  - `js/core/data_loader.js:614-617`
  - `js/ui/toolbar.js:2024-2025`
  - `js/ui/toolbar.js:4387-4400`
  - `js/ui/toolbar.js:4765-4770`
- 对照标尺：
  - Vercel Web Interface Guidelines 明确要求“URL reflects state”，尤其是 filters、tabs、expanded panels。

### 2.5 benchmark 对照
- ArcGIS Pro 的 pane set 能打开一组 pane 并关闭另一组 pane，本质上就是避免“所有面板同时常驻”。
- Mapbox Studio 的密度虽然高，但它把 `component` 和 `layer` 的层级关系说得很清楚；高复杂度不是原罪，层级不清才是。
- 相关来源：
  - ArcGIS Pro pane sets: https://pro.arcgis.com/en/pro-app/3.5/get-started/pane-sets.htm
  - Mapbox Studio manual: https://docs.mapbox.com/studio-manual/guides/
  - Mapbox components: https://docs.mapbox.com/studio-manual/guides/components/

## 主题三：字体序列和视觉语言分裂，阅读扫描路径不顺

### 3.1 标题和正文的字族本身没有拉开
- 全局 token 里 `--font-title` 和 `--font-body` 是同一套字族。
- 这意味着系统一开始就没有通过字族把“标题层”和“内容层”强行分开。
- 证据：
  - `css/style.css:33`
  - `css/style.css:34`

### 3.2 局部又突然插入另一套字族
- 语言按钮单独切到 `IBM Plex Sans`，会让局部控件看起来像来自另一套系统。
- 这种局部跳轨不会带来清晰层级，反而会制造“为什么这里长得不一样”的杂讯。
- 证据：
  - `css/style.css:7305`
  - `index.html:1283-1298`
  - 截图：`.runtime/browser/manual-audit/zoom-controls-steady.png`

### 3.3 同层级标题其实混成了至少三种语言
- `.section-header`：0.88rem / 700 / 0.01em
- `.section-header-block`：0.76rem / 700 / 0.05em
- boot eyebrow / shell heading / workbench kicker 一类又反复用 0.12em-0.14em 的大写追踪字距
- 这些元素在视觉上都在扮演“标题/标签”的角色，但不是同一种 typographic grammar。
- 证据：
  - `css/style.css:182-207`
  - `css/style.css:377-387`
  - `css/style.css:1609`
  - `css/style.css:3517`
  - `css/style.css:4215`
  - `css/style.css:4659`

### 3.4 细字、小胶囊、浅色标签被用得太多
- scenario context bar、zoom 区、右栏小标题、左栏 kicker 都在大量使用小号、低对比、短标签。
- 单个看很精致，但组合在一起时，真正需要优先读的内容并没有被拉得足够开。
- 证据：
  - 截图：`.runtime/browser/manual-audit/scenario-context-bar-steady.png`
  - 截图：`.runtime/browser/manual-audit/right-sidebar-steady.png`
  - 截图：`.runtime/browser/manual-audit/left-sidebar-steady.png`
  - 样式：`css/style.css:5142-5189`
  - 样式：`css/style.css:7294-7312`

### 3.5 字距和行高使用过于离散
- 当前样式里可见 `60` 处 `letter-spacing`、`79` 处 `line-height` 定义。
- 这不自动等于坏，但在当前界面里已经对应到很多不同的标题/标签/按钮/浮层系统，说明 typographic scale 没有足够收口。

## 主题四：控件语法与反馈语言不统一，用户难以快速读懂动作类型

### 4.1 首屏同时混用太多按钮方言
- 面板切换按钮
- 顶部 context chip 风格按钮
- 右上 zoom pill
- icon-only tool button
- segmented icon button
- 主行动 icon button
- Inspector tab button
- `?` 式上下文按钮
- 这些按钮都在首屏，但外形差异大于角色差异，用户不容易一眼判断它们是导航、模式切换还是立即执行。
- 证据：
  - `index.html:1221-1224`
  - `index.html:1261`
  - `index.html:1280-1307`
  - `index.html:1318-1488`
  - `index.html:1498-1512`
  - `css/style.css:4796-4847`
  - `css/style.css:5031-5061`

### 4.2 底部 dock 把多种语义硬压进一条带
- 工具切换、模式切换、编辑规则、地图动作、历史、颜色都在同一行同一套容器里。
- 视觉上它们是并列组，但语义上并不处于同一优先级。
- 用户的操作顺序感因此变差，更像在一个大工具盘里找按钮。
- 证据：
  - `index.html:1312-1489`
  - `css/style.css:4982-5138`
  - 截图：`.runtime/browser/manual-audit/bottom-dock-steady.png`

### 4.3 右侧 Inspector 的默认状态偏“静态展示”
- 默认首屏里，右栏虽然拥有 tab、搜索和多组 section，但当前焦点并不强。
- 它更像一个安静的资料柜，而不是“当前正在编辑谁、下一步要去哪”的主操作面板。
- 证据：
  - 截图：`.runtime/browser/manual-audit/right-sidebar-steady.png`
  - `index.html:1528-1584`
  - `index.html:1542-1833`

### 4.4 反馈系统分散在多个区域
- onboarding hint 在地图左上附近
- tool HUD chip 独立漂浮
- startup readonly banner 固定底部
- toast 在独立 viewport
- scenario context bar 本身又在表达当前状态
- 这会让“当前系统到底在告诉我什么”分散到不同地点，不是一个统一的反馈层。
- 证据：
  - `index.html:1229-1279`
  - `index.html:2073-2077`
  - `css/style.css:218-248`
  - `css/style.css:5142-5189`

### 4.5 guideline 交叉结论
- 正面项：
  - 有 skip link
  - 有 `:focus-visible`
  - 多个异步区用了 `aria-live`
  - 有 `prefers-reduced-motion`
- 主要问题不在无障碍底线，而在更高层的界面语法收口：
  - URL 没有承接主要 UI 状态
  - 首屏并存过多控制层
  - 按钮与浮层角色不够一眼可辨

## 外部标尺摘要

### ArcGIS Pro
- 关键观察：把 `ribbon`、`views`、`panes` 明确拆开；pane set 能按任务切换面板组合。
- 与本项目最相关的审查标尺：主工作区和上下文面板是否被明确区分。
- 来源：
  - https://pro.arcgis.com/en/pro-app/3.4/get-started/introducing-arcgis-pro.htm
  - https://pro.arcgis.com/en/pro-app/3.5/get-started/pane-sets.htm

### QGIS
- 关键观察：面板多并不违规，但必须允许快速退回 map canvas。
- 与本项目最相关的审查标尺：地图是否随时能重新成为唯一视觉中心。
- 来源：
  - https://docs.qgis.org/3.10/en/docs/user_manual/introduction/qgis_gui.html

### Mapbox Studio
- 关键观察：高复杂度界面依赖清晰的层级名义；`component` 可以包含很多 `layer`，但用户始终知道自己在编辑哪一层。
- 与本项目最相关的审查标尺：高密度控制是否仍然有明确层级归属。
- 来源：
  - https://docs.mapbox.com/studio-manual/guides/
  - https://docs.mapbox.com/studio-manual/guides/components/

### Victoria 3
- 关键观察：视觉方向可以有装饰感，但不能太重；重点是让交互目标有明确的 importance order。
- 与本项目最相关的审查标尺：风格存在感是否压过了决策优先级。
- 来源：
  - https://www.paradoxinteractive.com/games/victoria-3/news/dev-diary-49-graphic-overview

### Terra Invicta
- 关键观察：当信息密度高到一定程度，UI overhaul 会优先落在 readability、ledger、resource tracking、data table 上，而不是单纯做按钮美化。
- 与本项目最相关的审查标尺：高信息密度下是否还能保持读表效率与状态可读性。
- 来源：
  - https://store.steampowered.com/news/posts/?enddate=1737414408&feed=steam_community_announcements

## 总结结论
- 当前问题不是“某些控件不好看”，而是单页主壳同时承载了太多层级、太多控件语法、太多临时层。
- 你感受到的风格不统一、字体序列不稳、首页元素堆砌、交互层级过多、视觉重心不足，已经能被浏览器实景、静态结构和外部 benchmark 三条证据线同时确认。
- 最核心的判断可以压缩成一句话：
  - 当前界面不是不精致，而是系统太多、语法太多、重心太散。
